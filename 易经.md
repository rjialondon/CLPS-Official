## 一、八卦编码（道·码操作码层）

底爻为 MSB，对应 8 个物理电压级（现阶段用二进制模拟）。

| 卦名 | 卦象 | 三爻 | 二进制 | 电压（未来） | CLPS 操作码 | 卦义 |
|------|------|------|--------|------------|------------|------|
| 乾 ☰ | ䷀ | ═══ | 111 | 2.90V | `qian`  | 创生，资源初始化 |
| 兑 ☱ | ䷹ | ══╌ | 110 | 2.50V | `dui`   | 开口，输出，悦 |
| 离 ☲ | ䷝ | ═╌═ | 101 | 2.10V | `li`    | 附丽，引用，明 |
| 震 ☳ | ䷲ | ╌══ | 100 | 1.70V | `zhen`  | 动，运算，推进 |
| 巽 ☴ | ䷸ | ══╌ | 011 | 1.30V | `xun`   | 入，遍历，渗透 |
| 坎 ☵ | ䷜ | ╌═╌ | 010 | 0.90V | `kan`   | 险，安全运算 |
| 艮 ☶ | ䷳ | ╌╌═ | 001 | 0.50V | `gen`   | 止，锁，静 |
| 坤 ☷ | ䷁ | ╌╌╌ | 000 | 0.00V | `kun`   | 归元，释放 |

> 卦象读法：═ = 阳爻，╌ = 阴爻；从下往上读（底爻 = bit2 = MSB）。
> 电压值为未来多电平硬件的理论映射，现阶段 3 位二进制模拟。

## 六、两条宪法级原则（2026-02-26 确立）

### 原则一：底层涌现，向上映射

爻是底层，卦象从爻中涌现，不是反过来。
系辞曰："易者，象也。象也者，像也。"——象不是设计出来的，是从底层情形里像出来的。

**对应 CLPS 结构：**
```
底层代码（最多，最密）
    ↓ 自然映射出
阶段性目的（结界/块级意图）
    ↓ 总和凝结成
大纲（文件级脉络）
```
验证方向可以反向走（大纲展开→阶段逻辑通→底层通），
但**源头永远在底层**。大纲是底层长出来的，不是设计好再填代码的。

结界不是人为划定的边界，是底层代码密度自然凝聚的节点。
一个结界的名字，必须能被它包含的底层语句完整兑现——兑现不了，即为撒谎。

### 原则二：表里如一即合法，乙方能否承受是乙方的问题

**讼卦(6)·有孚** — 讼卦不审判诉求是否正义，只审判是否有孚（诚信、表里如一）。有孚，案子成立。赢不赢是另一回事。

**师卦(7)·贞** — 军队的合法性来自"贞"（正直、真实），不来自出师的理由。对方能否抵御，是对方的兵法问题。

**五行克制·中立** — 水克火，不是水的错，也不是道德问题。克制关系是自然规律，易经从不把克制本身判为非法。

**对应 CLPS 判定标准：**
- CLPS 只管甲方程序是否**表里如一**（底层真实映射出表象）
- 目的本身不在审查范围——渗透、加密、对抗、通信、插入外部设备，只要诚实呈现，即为合法
- 乙方的范围没有边界：文件系统、目标网络、对端协议栈、OS驱动、硬件控制器——凡程序之外者，皆乙方
- 防火墙拦不拦、驱动准不准、对端接不接，是乙方边界工程，不是甲方程序的责任
- 把"安全"与"诚实"分开：诚实是可验证的（1/10算力），安全是乙方的边界工程

**所有外部信息进入途径（甲方必须表里如一声明）：**

```
物理层：
    外部设备     → USB / 串口 / GPIO / 蓝牙 / NFC / RFID
    传感器       → 摄像头 / 麦克风 / GPS / 加速度计 / 温度
    网卡         → 网络接口 / 无线电

OS层：
    文件系统     → 读写 / 目录监听
    环境变量     → 进程启动时注入
    进程信号     → SIGINT / SIGTERM / 自定义信号
    共享内存     → mmap / 跨进程内存区
    管道         → pipe / FIFO / 消息队列

协议层：
    网络通信     → TCP / UDP / HTTP / WebSocket / 原始套接字
    广播/多播    → 局域网广播 / 组播
    DNS          → 域名解析作为信息通道

应用层：
    标准输入     → stdin / 键盘 / 触控 / 鼠标
    剪贴板       → 系统剪贴板注入
    API响应      → 外部服务返回值

时间/随机：
    时钟         → 系统时间 / 定时器触发
    随机数源     → 软件PRNG / 硬件熵源

人：
    对话/指令    → 任何想与系统交互的主体
```

**无论哪条途径，原则不变：**
甲方必须表里如一地呈现"我在用这条路径，目的是什么"。
乙方（文件系统、驱动、对端、OS、人）决定是否响应，是否放行。

**关于人作为乙方：**
想与系统交互，就得让系统知道你的目的。
系统愿不愿意对接，是系统自己的事。
大不了开一个子系统（家人卦·外部调取模型）与之对接，
交互结束，坤·归元，子系统散去，主系统不受污染。

> 易经不是答案的集合，是结构的集合。
> 六十四卦是六十四种情形的骨架——任何领域里，只要问题有真实的结构，卦象就在等它。
> 道·码用三千年前的骨架，长今天的代码。骨架是通的。
```

## 七、根基验证方法（从原则一具象化）

> 原则一「底层涌现，向上映射」在验证层面的应用：
> 引用易经概念时，优先从字源（说文解字）和考古实物出发，
> 而不仅依赖宋代以来的通行解释。底层锚定繁体原文，上层可用简体映射。

验证优先级：
1. **字源实证**：说文解字原文（有行号可核查）、甲骨文字形
2. **考古实物**：出土文物（如绳索纹陶器、亚形墓葬）
3. **先秦文献**：系辞传、说卦传、淮南子等
4. **汉代经典**：礼记、黄帝内经、京氏易传等
5. **宋代及后**：朱熹本义、邵雍先天图等（需标注为"通行版本"，不等于原始版本）

繁简原则：
- **宪法层推导**：锁定繁体（亞的隅角、圖的啚、龜的形态，简化后信息损失不可逆）
- **代码层使用**：可用简体（八卦关键字乾坤震巽坎离艮兑未被简化，无损）
